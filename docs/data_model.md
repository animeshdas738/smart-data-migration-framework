# Smart Data Migration Framework — Data Model

> Salesforce Technical Architecture · API v66.0 · No namespace

---

## Design Principles

Every object in this model exists to eliminate one class of migration failure. The mapping is deliberate:

| Failure mode | Object that prevents it |
|---|---|
| Silent data loss | `MigrationRowError__c` — every bad row is named, not silently dropped |
| Broken relationships | `MigrationRecordLog__c` (ID map role) + `MigrationObjectDependency__c` |
| No recovery on failure | `MigrationBatch__c` checkpoint + `MigrationJob__c` resume cursor |
| No visibility | `MigrationBatch__c` counters → LWC aggregation |
| No undo | `MigrationRecordLog__c` (rollback log role) + `MigrationRollback__c` |
| Data quality discovered too late | `MigrationRowError__c` Phase=PreValidation + dry-run flag on job |

---

## Entity Relationship Overview

```
MigrationJob__c
 ├─── MigrationObjectConfig__c (1..N, MasterDetail)
 │     ├─── MigrationObjectDependency__c (parent side, Lookup)
 │     ├─── MigrationObjectDependency__c (child side, Lookup)
 │     ├─── MigrationFieldMapping__c (1..N, MasterDetail)
 │     └─── MigrationBatch__c (1..N, MasterDetail)
 │           └─── MigrationRowError__c (0..N, MasterDetail)
 ├─── MigrationRecordLog__c (0..N, MasterDetail)   ← rollback log + ID map
 └─── MigrationRollback__c (0..1, MasterDetail)
```

## Objects

### 1. `MigrationJob__c` — Top-level job container

One record per migration run. Owns all state needed to resume, roll back, or report on the entire job.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `Name` | Auto Number | — | `MIG-{YYYY}-{0000}` |
| `Status__c` | Picklist | Required | `Draft` · `Validating` · `ReadyToLoad` · `Running` · `Paused` · `Completed` · `Failed` · `RolledBack` |
| `Mode__c` | Picklist | Required | `DryRun` · `Insert` · `Upsert` |
| `SourceType__c` | Picklist | Required | `CSV` · `LegacyCRM` · `REST_API` · `SalesforceOrg` |
| `BatchSize__c` | Number(3,0) | Default 200 | Hard cap at 200; Salesforce DML limit per transaction |
| `TotalRecords__c` | Number(18,0) | — | Sum across all object configs; set at job start |
| `LoadedCount__c` | Number(18,0) | Default 0 | Incremented by each successful batch |
| `FailedCount__c` | Number(18,0) | Default 0 | Incremented per DML-phase error |
| `SkippedCount__c` | Number(18,0) | Default 0 | Rows intentionally skipped (e.g. duplicates in dry-run) |
| `PercentComplete__c` | Formula(Percent) | — | `(LoadedCount__c / TotalRecords__c) * 100` |
| `StartedAt__c` | DateTime | — | Set when Status transitions to Running |
| `CompletedAt__c` | DateTime | — | Set on terminal status |
| `EstimatedCompletionAt__c` | DateTime | — | Recalculated after each batch: `StartedAt + (elapsed / PercentComplete) * 100` |
| `LastCompletedBatchSequence__c` | Number(8,0) | Default 0 | Global sequence of the last fully-committed batch; resume cursor |
| `IsDryRun__c` | Checkbox | Default false | When true: validation engine runs, zero DML committed |
| `AllowPartialSuccess__c` | Checkbox | Default true | Maps to `allOrNone = false` in Database DML calls |
| `ConfigJSON__c` | LongTextArea(32768) | — | Extended config (retry policy, rate limits, source credentials hash) |
| `ValidationSummary__c` | LongTextArea(32768) | — | Human-readable pre-load error report generated by dry-run |
| `RollbackStatus__c` | Picklist | Default `NotStarted` | `NotStarted` · `InProgress` · `Completed` · `Failed` |

**Indexes:** `Status__c` (for dashboard queries), `StartedAt__c` (for recency sorting).

### 2. `MigrationObjectConfig__c` — Per-object migration settings

One record per Salesforce object type within a job. Drives load order, field mapping, and per-object counters.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `MigrationJob__c` | MasterDetail | Required | Parent job |
| `Name` | Text(80) | Required | e.g. `Account`, `Contact`, `Revenue__c` |
| `ObjectApiName__c` | Text(80) | Required | Exact Salesforce API name |
| `ExternalIdField__c` | Text(80) | — | API name of the upsert key field (required when Mode = Upsert) |
| `LoadOrder__c` | Number(3,0) | Required | Topological sort output; lower loads first. 1 = no parents |
| `Mode__c` | Picklist | — | `Insert` · `Upsert` · `Update`; overrides job-level Mode when set |
| `SourceRecordCount__c` | Number(18,0) | — | Row count from source extract; set before job starts |
| `LoadedCount__c` | Number(18,0) | Default 0 | |
| `FailedCount__c` | Number(18,0) | Default 0 | |
| `SkippedCount__c` | Number(18,0) | Default 0 | |
| `Status__c` | Picklist | Default `Pending` | `Pending` · `InProgress` · `Completed` · `Failed` · `Skipped` |
| `HasSelfReference__c` | Checkbox | Default false | Triggers two-pass load strategy |
| `SelfReferenceField__c` | Text(80) | — | API name of the self-lookup (e.g. `ReportsToId`) |
| `RecordTypeHandling__c` | Picklist | Default `DeveloperName` | `DeveloperName` · `StaticId` · `None`; how RecordType is resolved |
| `AllowPartialSuccess__c` | Checkbox | Default true | Object-level override of job setting |

**Indexes:** `MigrationJob__c` (master-detail, auto-indexed), `LoadOrder__c` (sort for dependency resolution).

### 3. `MigrationObjectDependency__c` — Dependency graph edges

Explicit parent-child relationships between objects being migrated. One record per directed edge in the dependency graph. Populated by the relationship resolver at job configuration time (either auto-detected from Salesforce metadata or manually configured).

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `MigrationJob__c` | Lookup | Required, Indexed | Denormalized for job-scoped queries |
| `ParentObjectConfig__c` | MasterDetail | Required | The object that must load first |
| `ChildObjectConfig__c` | Lookup | Required | The object with the foreign key |
| `RelationshipField__c` | Text(80) | Required | API name of the lookup/MD field on the child object (e.g. `AccountId`) |
| `DependencyType__c` | Picklist | Required | `MasterDetail` · `Lookup` · `SelfReference` |
| `IsAutoDetected__c` | Checkbox | Default false | True = derived from Salesforce Describe; false = manually added |

**Design note:** Self-referential relationships (e.g., `Contact.ReportsToId → Contact`) are represented with the same object on both sides. The relationship resolver detects this and sets `HasSelfReference__c = true` on the ObjectConfig rather than creating a dependency edge (a self-edge would create a topological cycle).

### 4. `MigrationFieldMapping__c` — Source-to-target field mappings

One record per column being migrated for a given object. The validation engine and bulk load engine both read these at job start and cache them in instance variables — never per-row SOQL.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `MigrationObjectConfig__c` | MasterDetail | Required | Parent object config |
| `SourceFieldName__c` | Text(255) | Required | Column name in source (CSV header, API JSON key, etc.) |
| `TargetFieldApiName__c` | Text(80) | Required | Salesforce field API name |
| `TargetFieldType__c` | Picklist | — | `Text` · `Number` · `Date` · `DateTime` · `Boolean` · `Picklist` · `MultiPicklist` · `Lookup` · `Email` · `Phone` · `URL` · `Currency` · `TextArea`; copied from Describe at config time |
| `TargetFieldLength__c` | Number(6,0) | — | Max character length from Describe; used by schema validator |
| `TransformationType__c` | Picklist | Default `Direct` | `Direct` · `StaticValue` · `DateFormat` · `PicklistMap` · `Concatenate` · `Truncate` · `Nullify` · `ExternalIdLookup` |
| `TransformationConfig__c` | LongTextArea(4096) | — | JSON params. E.g. for PicklistMap: `{"New":"New_Lead","Active":"Active__c"}`; for DateFormat: `{"from":"MM/DD/YYYY","to":"YYYY-MM-DD"}` |
| `IsExternalIdField__c` | Checkbox | Default false | True = this is the upsert key; validated as unique |
| `IsParentLookup__c` | Checkbox | Default false | True = this field holds a legacy parent ID that must be resolved via `MigrationRecordLog__c` before DML |
| `ParentObjectApiName__c` | Text(80) | — | Required when `IsParentLookup__c = true`; identifies which parent's log to query |
| `IsRequired__c` | Checkbox | Default false | Pre-validation raises error if value is null or empty |
| `DefaultValue__c` | Text(255) | — | Applied when source value is blank; applied before required check |
| `SequenceOrder__c` | Number(3,0) | Default 1 | Processing order; used for Concatenate transforms |
| `IsActive__c` | Checkbox | Default true | Inactive mappings are ignored at runtime |

**Design note:** `ExternalIdLookup` transformation type signals the bulk load engine to replace the source legacy ID with the live Salesforce ID from `MigrationRecordLog__c` (via Platform Cache). This is the bridge between the relationship resolver and the field-level mapping.

### 5. `MigrationBatch__c` — Individual batch execution

One record per 200-row chunk per object. The backbone of resumability and the primary source for the progress dashboard.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `MigrationObjectConfig__c` | MasterDetail | Required | Parent object config |
| `MigrationJob__c` | Lookup | Required, Indexed | Denormalized; enables dashboard SOQL without traversing two hops |
| `Name` | Auto Number | — | `BATCH-{000000}` |
| `BatchNumber__c` | Number(5,0) | Required | Sequence within this object (1-based) |
| `GlobalBatchSequence__c` | Number(8,0) | Required | Monotonically increasing across all objects in the job; used as resume cursor |
| `Status__c` | Picklist | Default `Pending` | `Pending` · `Running` · `Success` · `PartialSuccess` · `Failed` · `Skipped` · `RolledBack` |
| `AttemptedCount__c` | Number(4,0) | Default 0 | Rows sent to DML (0 in dry-run) |
| `SuccessCount__c` | Number(4,0) | Default 0 | Rows where `Database.SaveResult.isSuccess() = true` |
| `FailureCount__c` | Number(4,0) | Default 0 | Rows with DML errors |
| `PreValidationErrorCount__c` | Number(4,0) | Default 0 | Rows rejected before DML (schema/required/picklist failures) |
| `SourceRowStart__c` | Number(18,0) | Required | 0-based offset in the source dataset |
| `SourceRowEnd__c` | Number(18,0) | Required | Inclusive end offset |
| `StartedAt__c` | DateTime | — | |
| `CompletedAt__c` | DateTime | — | |
| `DurationSeconds__c` | Formula(Number) | — | `(CompletedAt__c - StartedAt__c) * 86400` |
| `ApexJobId__c` | Text(18) | — | `AsyncApexJob.Id` for this batch's Queueable/Batchable invocation |
| `IsCheckpoint__c` | Checkbox | Default false | True = this is the most recently committed batch; the resume marker |
| `RetryCount__c` | Number(2,0) | Default 0 | Incremented on retry; halt after threshold from `MigrationFrameworkConfig__mdt` |
| `IsSecondPass__c` | Checkbox | Default false | True = self-referential field backfill pass |
| `BatchError__c` | LongTextArea(4096) | — | Governor-level or framework error (not row-level); e.g. heap exceeded |

**Indexes:** `MigrationJob__c` (for dashboard), `GlobalBatchSequence__c` (for resume cursor lookup), `IsCheckpoint__c` (filtered query to find resume point).

### 6. `MigrationRowError__c` — Row-level errors

One record per field-level problem, per row. Covers both the pre-validation phase (before any DML) and DML-phase partial failures. This is the source of the "field-by-field, row-by-row" error report.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `MigrationBatch__c` | MasterDetail | Required | Parent batch |
| `MigrationJob__c` | Lookup | Required, Indexed | Denormalized for job-wide error reports |
| `Phase__c` | Picklist | Required | `PreValidation` · `DML` |
| `RowNumber__c` | Number(18,0) | Required | Absolute row number in source dataset |
| `SourceExternalId__c` | Text(255) | — | The row's external/legacy identifier from source |
| `ObjectApiName__c` | Text(80) | Required | Denormalized for filtering without joining to batch |
| `FieldApiName__c` | Text(80) | — | The specific field that failed; null for row-level DML errors |
| `ErrorType__c` | Picklist | Required | `RequiredFieldMissing` · `FieldTooLong` · `InvalidPicklistValue` · `InvalidFieldType` · `DuplicateExternalId` · `ParentRecordNotFound` · `RecordTypeNotFound` · `DMLError` · `UnmappedField` |
| `ErrorMessage__c` | LongTextArea(4096) | Required | Full Salesforce error string or framework-generated message |
| `SourceValue__c` | Text(255) | — | The offending value; truncated to 255 chars for display |
| `SuggestedFix__c` | LongTextArea(1000) | — | Framework-generated remediation hint (e.g. "Allowed picklist values: Active, Inactive, Pending") |
| `IsResolved__c` | Checkbox | Default false | Client marks resolved after fixing source data; used to filter the re-run scope |

**Indexes:** `MigrationJob__c` (error report queries), `Phase__c` (filter pre-validation vs DML), `ErrorType__c` (grouped error summary).

**Volume note:** Expect 0–10% error rate. For a 100k record migration: up to 10,000 error records. Custom objects handle this comfortably. Rows marked `IsResolved__c = true` can be archived after source data is corrected.

### 7. `MigrationRecordLog__c` — Rollback log and ID map

Dual-purpose: every record written to Salesforce during the migration is logged here. The rollback engine deletes from this log in reverse `LoadOrder__c` (children before parents). The relationship resolver queries this log to translate legacy parent IDs into live Salesforce IDs for child rows.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `MigrationJob__c` | MasterDetail | Required | Parent job |
| `MigrationBatch__c` | Lookup | Required, Indexed | Which batch produced this record |
| `ObjectApiName__c` | Text(80) | Required, Indexed | Salesforce object type |
| `SalesforceId__c` | Text(18) | Required | The ID of the inserted/upserted record |
| `LegacyId__c` | Text(255) | Required, External ID | Source system identifier; indexed for FK resolution lookup |
| `ExternalIdValue__c` | Text(255) | — | Value of the upsert key field (may differ from LegacyId in format) |
| `Action__c` | Picklist | Required | `Inserted` · `Updated` · `Upserted` |
| `LoadOrder__c` | Number(3,0) | Required | Copied from `MigrationObjectConfig__c.LoadOrder__c`; governs delete sequence during rollback |
| `WrittenAt__c` | DateTime | Required | Timestamp of the DML commit |
| `RollbackStatus__c` | Picklist | Default `Active` | `Active` · `Deleted` · `DeleteFailed` |

**Indexes:** `LegacyId__c` (External ID, indexed — FK resolution), `MigrationJob__c` (master-detail, auto-indexed), `ObjectApiName__c + MigrationJob__c` (combined filter for ID resolution; see Platform Cache section), `LoadOrder__c` (rollback delete ordering).

**Volume note:** 1 record per successfully migrated source row. For 100k records this is ~100k log records. Post-migration cleanup job should archive or delete this data once the rollback window has closed (typically 30 days post go-live). Do not delete during rollback; mark `RollbackStatus__c = Deleted` instead to preserve the audit trail.

**ID resolution in Apex (critical pattern):**
```apex
// Query once at batch start, cache in instance variable — never per-row
Map<String, Id> parentIdMap = new Map<String, Id>();
for (MigrationRecordLog__c log : [
    SELECT LegacyId__c, SalesforceId__c
    FROM MigrationRecordLog__c
    WHERE MigrationJob__c = :jobId
      AND ObjectApiName__c = :parentObjectName
      AND RollbackStatus__c = 'Active'
]) {
    parentIdMap.put(log.LegacyId__c, log.SalesforceId__c);
}
// Also write to Platform Cache for cross-transaction access
```

## Governor Limit Architecture

Salesforce's governor limits shape every architectural decision. These are the critical constraints and how the model addresses them.

### DML Rows per Transaction — 10,000 limit

`BatchSize__c` is capped at 200 to leave headroom for trigger-generated DML (workflow updates, process builder actions, platform event publishes). Each Queueable invocation processes exactly one `MigrationBatch__c` record.

Row error inserts (`MigrationRowError__c`) and record log inserts (`MigrationRecordLog__c`) are batched with the main DML. A 200-row batch could generate up to 400 DML rows (200 target records + 200 log entries) — well inside the 10k limit.

### SOQL Queries per Transaction — 200 limit (async)

The Apex batch executor queries the following at start():
- 1 query: `MigrationJob__c` (job config + mode)
- 1 query: `MigrationObjectConfig__c` + `MigrationFieldMapping__c` (nested subquery, counts as 2)
- 1 query: `MigrationRecordLog__c` for parent ID map (per parent object type)
- 1 query: `MigrationBatch__c` (current batch record)

Total: ~6 queries per batch executor invocation, leaving 194 for trigger chains.

### Heap Size — 12MB (async)

Parent ID maps are the primary heap consumer. For 100k parent records: `Map<String, Id>` ≈ 100k × (255 + 18) bytes ≈ 27MB — exceeds the limit.

**Solution: Platform Cache.** ID maps are written to the configured partition after each parent object batch completes. Child batches read from cache, falling back to SOQL only on a cache miss. With Platform Cache enabled, heap usage per batch stays under 2MB.

```
Platform Cache key pattern:  {jobId}_{objectApiName}
Value:                        Serialized Map<String, Id> (JSON)
TTL:                          3600 seconds (1 hour; reset on each write)
```

If `EnablePlatformCache__c = false`, the framework falls back to chunked SOQL with `OFFSET` paging, which is slower but governor-safe.

## Key Query Patterns

The following SOQL patterns drive the framework at runtime. All leverage the indexed fields defined above.

**Dashboard: live counters**
```sql
SELECT SUM(SuccessCount__c), SUM(FailureCount__c), COUNT(Id)
FROM MigrationBatch__c
WHERE MigrationJob__c = :jobId
  AND Status__c IN ('Success', 'PartialSuccess')
```

**Resume: find restart point**
```sql
SELECT GlobalBatchSequence__c, MigrationObjectConfig__c
FROM MigrationBatch__c
WHERE MigrationJob__c = :jobId
  AND IsCheckpoint__c = true
ORDER BY GlobalBatchSequence__c DESC
LIMIT 1
```

**Validation report: all pre-load errors for a job**
```sql
SELECT ObjectApiName__c, FieldApiName__c, ErrorType__c,
       RowNumber__c, SourceValue__c, SuggestedFix__c, ErrorMessage__c
FROM MigrationRowError__c
WHERE MigrationJob__c = :jobId
  AND Phase__c = 'PreValidation'
ORDER BY RowNumber__c ASC
```

**Rollback: objects to delete in reverse order**
```sql
SELECT SalesforceId__c, ObjectApiName__c, LoadOrder__c
FROM MigrationRecordLog__c
WHERE MigrationJob__c = :jobId
  AND RollbackStatus__c = 'Active'
ORDER BY LoadOrder__c DESC
```

**ID resolution: translate legacy parent ID to SF ID**
```sql
SELECT LegacyId__c, SalesforceId__c
FROM MigrationRecordLog__c
WHERE MigrationJob__c = :jobId
  AND ObjectApiName__c = :parentObject
  AND RollbackStatus__c = 'Active'
```

## Data Lifecycle

```
Job created (Draft)
    │
    ▼
Field mappings + object configs populated
    │
    ▼
Dependency resolver runs → sets LoadOrder__c on each ObjectConfig
    │                        creates MigrationObjectDependency__c edges
    ▼
Validation engine runs (dry-run or pre-flight)
    │   reads MigrationFieldMapping__c + live org Describe
    │   writes MigrationRowError__c (Phase=PreValidation)
    │   sets Job.Status = ReadyToLoad (0 errors) or stays Validating (errors exist)
    ▼
Bulk load engine processes objects in LoadOrder__c sequence
    │   For each 200-row batch:
    │     1. Reads MigrationFieldMapping__c (from cache)
    │     2. Resolves parent IDs via MigrationRecordLog__c / Platform Cache
    │     3. Applies field transformations
    │     4. Runs schema re-validation (live Describe check)
    │     5. Calls Database.upsert/insert (allOrNone=false)
    │     6. Writes MigrationRecordLog__c for successes
    │     7. Writes MigrationRowError__c for failures (Phase=DML)
    │     8. Updates MigrationBatch__c counters + IsCheckpoint__c
    │     9. Updates MigrationObjectConfig__c counters
    │    10. Updates MigrationJob__c counters
    │
    │   For self-referential objects:
    │     First pass: insert/upsert with null self-reference field
    │     Second pass: update self-reference field using now-available SF IDs
    │
    ▼
Job.Status = Completed (all batches done)
    │
    ▼ (optional rollback)
MigrationRollback__c created
Apex batch deletes MigrationRecordLog__c records in LoadOrder__c DESC order
Sets MigrationRecordLog__c.RollbackStatus__c = Deleted
Sets Job.Status = RolledBack
```

## Storage Estimation

| Object | Records per 100k migration | Est. size |
|---|---|---|
| `MigrationJob__c` | 1 | < 1 KB |
| `MigrationObjectConfig__c` | 5–15 | < 1 KB |
| `MigrationObjectDependency__c` | 5–20 | < 1 KB |
| `MigrationFieldMapping__c` | 50–300 | < 30 KB |
| `MigrationBatch__c` | ~500 | < 500 KB |
| `MigrationRowError__c` | 0–10,000 | < 5 MB |
| `MigrationRecordLog__c` | ~100,000 | ~100 MB |
| `MigrationRollback__c` | 0–1 | < 1 KB |

`MigrationRecordLog__c` is the dominant storage consumer. It is **temporary by design** — once the migration is confirmed successful and the rollback window closes (default 30 days, configured in `MigrationFrameworkConfig__mdt`), a scheduled Apex cleanup job deletes all log records for completed jobs. Provide a "Archive & Close" action on `MigrationJob__c` that triggers this cleanup.
